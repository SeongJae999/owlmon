01

모니터링이란 무엇인가

병원에서 환자에게 심전도 기계를 붙여놓는 것처럼, 서버에 모니터링 도구를 붙여두는 것입니다. 서버 상태를 계속 측정하고 이상 발생 시 담당자에게 알림을 보내며, 기록을 남겨 나중에 원인을 분석합니다.

❌
모니터링 없을 때
고객 "앱이 안 열려요"
→ 그제야 인지
→ 원인 파악
→ 해결
(고객이 먼저 발견)
✅
모니터링 있을 때
모니터링 도구가 이상 감지
→ 담당자에게 즉시 알림
→ 해결
(고객보다 먼저 발견)

서버 모니터링 항목 (사람 건강검진 비유)

건강검진모니터링설명
심박수CPU 사용률서버가 얼마나 바쁜지
혈압메모리 사용량작업 공간이 얼마나 남았는지
체온서버 온도과열 여부
혈당디스크 잔여 공간저장공간 부족 여부
호흡네트워크 트래픽데이터 송수신량
반사신경응답 시간 (Latency)요청에 얼마나 빨리 반응하는지
심전도에러율 (Error Rate)전체 요청 중 실패 비율

모니터링 vs 옵저버빌리티(Observability) NEW

최근 업계에서는 "모니터링"을 넘어 "옵저버빌리티"라는 개념이 대세입니다. 둘의 차이를 이해하는 것이 중요합니다.

📊 모니터링 (Monitoring)
  • 무엇이 고장났는지 알려줌
  • 미리 정한 지표를 추적
  • "CPU 95% 초과" 같은 알림
  • 알려진 문제에 대한 대응
🔍 옵저버빌리티 (Observability)
  • 왜 고장났는지 파악 가능
  • Metrics + Logs + Traces 통합
  • "이 API가 느린 이유는 DB 쿼리 N번선" 추적
  • 처음 보는 문제도 진단 가능
💡 모니터링은 옵저버빌리티의 부분집합입니다. 모니터링 → 옵저버빌리티로 진화하는 것이 현재 업계 흐름이며, OpenTelemetry가 그 핵심 표준입니다.
02

모니터링의 3가지 층

모니터링은 하나의 개념이 아니라 3가지 층이 있습니다. 도구마다 담당하는 층이 다릅니다.

APM 층 (Application Performance Monitoring)
"이 API가 왜 느려?" "어떤 쿼리가 문제야?" "사용자 요청이 어디서 병목?"
Jennifer · Sentry · SigNoz
인프라 모니터링 층 (Infrastructure)
"CPU가 90%야" "디스크 꽉 찼어" "네트워크 대역폭 포화"
Zabbix · Prometheus · Checkmk
로그 분석 층 (Log Management)
"어떤 에러가 났어?" "누가 접속했어?" "보안 이벤트 추적"
ELK Stack · Loki · Graylog

추가: 합성 모니터링 & 업타임 체크 NEW

위 3가지 층 외에 "외부에서 바라보는 시점"의 모니터링도 중요합니다.

🌐 업타임 모니터링
  • "우리 웹사이트가 지금 살아있나?"
  • 외부에서 HTTP 핑을 보내 확인
  • 대표 도구: Uptime Kuma, Gatus
🤖 합성 모니터링 (Synthetic)
  • "사용자처럼 흉내내서 테스트"
  • 로그인→주문→결제 시나리오 자동 실행
  • 대표 도구: Checkmk, Grafana Synthetic
💡 업종마다 필요한 층이 다릅니다. IT 개발팀 있는 회사 → APM 필수 / IT 전담 없는 중소기업 → 인프라 층 + 업타임 체크만으로 충분

기술 스택

구성요소언어이유
서버·에이전트C빠르고 가벼움, OS 직접 접근

🔔 알림 폭탄

같은 경고가 수백 개 쏟아져 진짜 중요한 알림을 놓침. 카카오톡 연동도 기본 미지원.

개선 가능
🖥 복잡한 영문 UI

메뉴가 영어로 20개 이상. 비전문가(공장 총무, 병원 원무과)는 화면만 봐도 포기.

개선 가능
📱 모바일 공식 앱 없음

퇴근 후 장애 발생 시 노트북을 켜야 확인 가능. 공식 모바일 앱이 없음.

개선 가능
📊 보고서 불편

기본 보고서는 그래프 이미지 나열. 한글 요약 없음, 인쇄 시 깨짐.

개선 가능
⚙ 초기 설정 복잡

서버 추가마다 에이전트 설치→등록→항목설정→임계값→알림 전부 수작업.

자동화 가능
🤔 원인을 안 알려줌

"CPU 95% 초과"만 알려주고 왜 그런지, 어떻게 해야 하는지는 알려주지 않음.

AI 접목 가능

# 장점
- 서버가 직접 가져가므로 "뭘 수집하는지" 중앙 제어 가능

모니터링 시스템 개발에 필수로 알아야 할 프로토콜과 기술 스택입니다.

SNMP (Simple Network Management Protocol)

1988년부터 사용된 네트워크 장비 모니터링의 표준 프로토콜입니다. 라우터, 스위치, 프린터, UPS 등 거의 모든 네트워크 장비가 지원합니다.

// SNMP_ARCHITECTURE.md
// 핵심 구성요소
SNMP Manager (NMS)  ← 중앙 모니터링 서버
SNMP Agent          ← 장비에 내장된 소프트웨어
MIB (Management Information Base) ← 수집 가능 항목 목록
OID (Object Identifier)  ← 각 항목의 고유 번호

// 동작 방식
1. Manager → Agent: GET 요청 (정보 요청)
2. Agent → Manager: Response (정보 응답)
3. Agent → Manager: TRAP (이벤트 알림, 비동기)

// 버전별 차이
v1:  평문 커뮤니티 스트링 (보안 취약)
v2c: GetBulk 추가 (효율성 개선), 보안은 여전히 약함
v3:  인증 + 암호화 지원 (권장)

// 수집 가능 메트릭
CPU 사용률, 메모리, 인터페이스 상태
대역폭 사용량, 에러 카운트, 온도
장비 업타임, 팬 속도, 전력 상태

OTLP (OpenTelemetry Protocol)

🔗 OTel의 데이터 전송 표준
  • gRPC 및 HTTP 둘 다 지원
  • Metrics, Logs, Traces 3가지 신호 모두 전송
  • Protobuf 기반 직렬화 → 높은 효율
  • 모든 주요 모니터링 백엔드가 OTLP 수신 지원
# OTLP 엔드포인트 예시
gRPC: localhost:4317
HTTP:  localhost:4318/v1/metrics
HTTP:  localhost:4318/v1/traces
HTTP:  localhost:4318/v1/logs

PromQL (Prometheus Query Language)

📐 메트릭 분석의 사실상 표준 쿼리 언어
# CPU 사용률 5분 평균
rate(node_cpu_seconds_total{mode="idle"}[5m])

# 메모리 사용률 (퍼센트)
(1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) * 100

# HTTP 요청 에러율
rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m]) * 100

# P99 응답시간 (히스토그램)
histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))

VictoriaMetrics, Grafana, Thanos 등 대부분의 도구가 PromQL 호환 쿼리를 지원하므로, PromQL을 한 번 익히면 여러 플랫폼에서 사용 가능합니다.

기술역할알아야 하는 이유
SNMP v2c/v3네트워크 장비 데이터 수집온프레미스 납품 시 라우터/스위치 모니터링 필수

Protocol Buffers고효율 직렬화대량 메트릭 전송 시 JSON보다 10x 효율

시계열 데이터베이스 (TSDB) NEW

모니터링 데이터는 "시간 + 값"의 연속입니다. 일반 DB(MySQL 등)로는 대량의 시계열 데이터를 효율적으로 처리할 수 없어, 전용 데이터베이스를 사용합니다.

// 시계열 데이터란?
// 일반 데이터 (RDBMS)
| user_id | name | email            |
| 1       | 김철수 | cs@example.com |

// 시계열 데이터 (TSDB)
| timestamp           | metric        | value | host    |
| 2026-03-14 10:00:00 | cpu_usage     | 45.2  | web-01  |
| 2026-03-14 10:00:10 | cpu_usage     | 47.8  | web-01  |
| 2026-03-14 10:00:20 | cpu_usage     | 52.1  | web-01  |
↑ 시간 축으로 계속 쌓이는 데이터
  쓰기가 매우 많고, 오래된 데이터는 집계 후 삭제

TSDB 비교

TSDB사용처장점단점
Prometheus TSDB Prometheus 내장 단순, 빠름, 로컬 스토리지 장기 보관 어려움, 단일 노드 한계
VictoriaMetrics Prometheus 장기 스토리지 높은 압축률, PromQL 호환 클러스터 모드는 유료
ClickHouse SigNoz, OpenObserve 등 초고속 분석 쿼리, SQL 지원, 고카디널리티에 강함 운영 복잡도, 리소스 요구 높음
InfluxDB IoT, 범용 시계열 전용 쿼리 언어(Flux), 쓰기 성능 우수 v3로 대전환 중, 라이선스 불투명
TimescaleDB PostgreSQL 확장 SQL 그대로 사용, PostgreSQL 생태계 활용 전용 TSDB 대비 쓰기 성능 낮음
💡 MVP 추천: 시작할 때는 Prometheus TSDB (내장)로 충분합니다. 데이터가 수개월 이상 쌓이거나 서버 100대 이상이면 VictoriaMetrics 또는 ClickHouse로 마이그레이션을 고려하세요.

카디널리티(Cardinality) 문제 — TSDB의 핵심 이슈

# 카디널리티 = 고유한 시계열(time series) 수

# 낮은 카디널리티 (괜찮음)
cpu_usage{host="web-01"}        // 서버 10대 = 시계열 10개
cpu_usage{host="web-02"}

# 높은 카디널리티 (위험!)
http_requests{user_id="12345"}  // 사용자 100만명 = 시계열 100만개
http_requests{user_id="12346"}  // 메모리 폭발, 쿼리 느려짐

→ Prometheus는 고카디널리티에 취약 (메모리 부족)
→ ClickHouse는 고카디널리티에도 강함 (컬럼 스토리지)

# 실무 규칙: 라벨(태그)에 사용자 ID, 세션 ID 같은
# 무한히 늘어나는 값을 넣지 말 것!

알림(Alerting) 시스템 설계 NEW

모니터링의 핵심 가치는 "문제를 빠르게 알려주는 것"입니다. 알림 시스템의 설계가 부실하면 아무리 데이터를 잘 수집해도 무용지물입니다.

알림 설계 4원칙

❌ 나쁜 알림
  • 같은 알림 100개 연속 발송 (알림 폭탄)
  • "CPU 80% 초과" → 5분 뒤 정상 → 반복
  • 새벽 3시에 "디스크 72% 사용" (긴급 아님)
  • 알림만 보내고 해결 방법 안 알려줌
✅ 좋은 알림
  • 중복 제거: 같은 문제는 1건만 발송
  • 디바운싱: 5분 이상 지속 시에만 발송
  • 심각도 분류: Critical → 즉시, Warning → 오전에
  • 원인 추정 + 해결 가이드 함께 제공

알림 파이프라인 설계

// 알림 처리 흐름

메트릭 수집 → 임계값 평가 → 디바운싱 → 중복 제거 → 심각도 분류 → 채널 라우팅
                                                                          ↓
                                                               ┌─────────────────┐
                                                               │ Critical → 전화  │
                                                               │ Warning  → 카톡  │
                                                               │ Info     → 슬랙  │
                                                               └─────────────────┘

// 디바운싱 (채터링 방지)
규칙: CPU > 90% 가 5분 이상 지속될 때만 알림
잠깐 스파이크 → 무시
지속적 과부하 → 알림 발송

// 중복 제거 (De-duplication)
동일 호스트 + 동일 메트릭 알림은 해소될 때까지 1건만
해소되면 "복구됨" 알림 발송

// 에스컬레이션
30분 미응답 → 다음 담당자에게 전달
1시간 미응답 → 팀장에게 전달

핵심 논문 & 문서 분석 RESEARCH

모니터링 시스템의 이론적 기반이 되는 핵심 논문과 문서들입니다. "왜 현재의 도구들이 이렇게 설계되었는가"를 이해하는 데 필수적입니다.

📄 논문 1: Google Dapper (2010)
항목내용
제목Dapper, a Large-Scale Distributed Systems Tracing Infrastructure
저자Benjamin H. Sigelman 외 (Google)
중요도★★★★★ 필독 — 분산 추적의 기원. OpenTelemetry의 직접적 조상

3가지 설계 목표

1. 낮은 오버헤드 — 추적이 서비스 성능에 미치는 영향 최소화
2. 애플리케이션 투명성 — 개발자가 추적 코드를 직접 작성 불필요
   → 핵심 라이브러리(RPC, 스레딩)에만 C++ ~1,500줄 계측
3. 대규모 확장성 — 샘플링(0.01~1%)으로 오버헤드 제어

핵심 개념: Span과 Trace

// Dapper가 만든 개념 — OTel에서 지금도 그대로 사용
Trace = 하나의 요청이 시스템을 통과하는 전체 경로 (트리 구조)
Span  = 하나의 서비스에서 처리되는 단위 작업
         Trace ID + Span ID + Parent Span ID 저장

// 예: 검색 요청 추적
[Trace: 검색 요청]
  └─ Span: 프론트엔드 (50ms)
       ├─ Span: 광고 서비스 (20ms)
       ├─ Span: 웹 검색 (120ms)
       │    └─ Span: DB 쿼리 (80ms) ← 병목!
       └─ Span: 이미지 검색 (30ms)
⚠️ Dapper → Zipkin(Twitter) → Jaeger(Uber) → OpenTracing → OpenTelemetry로 이어지는 계보. OTel의 구조를 이해하려면 이 논문이 출발점입니다.
📄 논문 2: Facebook Gorilla (2015)
항목내용
제목Gorilla: A Fast, Scalable, In-Memory Time Series Database
저자Tuomas Pelkonen 외 (Facebook)
출처VLDB 2015 (데이터베이스 분야 최고 학회)
중요도★★★★★ 필독 — 시계열 DB 압축의 교과서. Prometheus TSDB가 직접 채용

핵심: 12배 압축 알고리즘

// Facebook: 초당 1,200만 데이터 포인트, 20억 고유 시계열
// 기존 HBase 대비: 쿼리 지연 73배↓, 처리량 14배↑

데이터 포인트 = (타임스탬프 64bit + 값 64bit) = 16 bytes
압축 후 평균 1.37 bytes → 12배 압축!

// 타임스탬프: Delta of Delta 압축
30초 간격 수집 → 연속 delta 거의 동일 → delta의 delta 저장
96%가 0 → 1비트로 표현

// 값: XOR 압축
CPU 45.2% → 45.3% → 45.1% (조금씩 변화)
연속 값의 XOR → 대부분 0
59%가 이전과 동일 → 1비트로 표현

3가지 설계 원칙

  • "최근 데이터 > 오래된 데이터" — 장애 진단 시 최근 26시간이 핵심. 인메모리에 최근 데이터만 유지
  • "집계가 개별 포인트보다 중요" — "평균 추세"를 보지, "3시 14분 22초의 47.3%"를 보지 않음
  • "쓰기 가용성 최우선" — 모니터링 데이터는 유실되면 복구 불가. 항상 쓸 수 있어야 한다

4가지 골든 시그널

1. Latency (지연시간)  — 요청 처리 시간. p95/p99로 측정 (평균은 거짓말)
2. Traffic (트래픽)    — 시스템 부하량. 초당 요청 수(RPS)
3. Errors (에러율)    — 실패한 요청 비율. HTTP 5xx + 암시적 에러
4. Saturation (포화도) — 리소스 한계 접근도. CPU/메모리/디스크

// Google SRE 알림 원칙
"알림이 울릴 때마다 긴급 대응할 수 있어야 한다"
"자동으로 해결 가능한 문제는 알림이면 안 된다"
"하루에 몇 번 이상 긴급 대응하면 피로해진다"
💡 대시보드에 골든 시그널 4개를 기본 패널로 넣고, 알림은 이 4개 기반으로만 설정하면 80%의 장애를 커버할 수 있습니다.

개발 로드맵 & 학습 순서

기술 학습 순서 (보강판) ENHANCED

1
Uptime Kuma 설치 & 분석 — Docker 한 줄로 설치. 코드 구조(Node.js + Vue.js + SQLite)를 분석하면 모니터링 MVP의 교과서. 알림 채널 연동 방식도 참고.
2
Grafana + Prometheus 연동 — 30분 안에 예쁜 대시보드 완성. PromQL 기초 학습. 메트릭 수집-저장-시각화 파이프라인의 핵심 이해.
3
OpenTelemetry Collector 구축 — OTel Collector를 설치하고, 간단한 애플리케이션에서 메트릭/트레이스를 보내보기. 벤더 중립 파이프라인 이해.
4
Zabbix 학습 + API 활용 — 국내 납품용 핵심 도구. JSON-RPC API로 호스트 목록, 메트릭 조회, 알림 설정을 Python/JS에서 해보기.
5
간단한 에이전트 직접 만들기 — Python으로 CPU/메모리/디스크를 수집하여 REST API로 전송하는 에이전트 제작. Go로 리팩토링하면 실력 급상승.
6
카카오톡 알림 연동 — 알림톡 API 또는 카카오워크 Webhook으로 실제 알림 발송 구현. 디바운싱, 중복 제거 로직 추가.
7
실전 인터뷰 — 주변 IT 담당자 5명에게 "모니터링 관련해서 제일 불편한 게 뭐예요?" 공통된 불편함 하나 발견.
8
그 불편함 하나만 해결하는 도구 만들기 — 무료로 써달라고 하고 피드백 받기. 반복.

### ~~2순위 — 이상탐지~~ ✅ Phase 1 완료

#### 이상탐지 — 구현 완료 (Phase 1)

**Z-score + 이동평균** (`server/anomaly/detector.go`)
- 호스트별/메트릭별 순환 버퍼 기반 이동평균, 표준편차 (윈도우 120개 = 1시간)
- 계절성 보정: 업무시간(평일 09~18), 야간, 주말별 별도 통계 관리
- Z-score ≥ 3.0 → critical, ≥ 2.5 → warning
- 외부 라이브러리 zero, 순수 Go 구현

**선형회귀 디스크 예측** (`server/anomaly/predictor.go`)
- 최소제곱법 선형회귀 (6시간 윈도우, 30초 간격)
- R² (결정계수)로 모델 신뢰도 평가 — R² < 0.5이면 알림 미발송
- 7일 이내 고갈 예상 시 알림

**API**: `GET /api/anomaly` (이상 + 예측 통합), `GET /api/anomaly/disk` (디스크만)
**UI**: AnomalyPanel 컴포넌트 — 상세 보기에서 이상 항목 + 디스크 예측 카드 표시

#### 이상탐지 — 향후 Phase 2/3

| Phase | 기술 | 목적 | 상태 |
|-------|------|------|------|
| 2 | Isolation Forest | CPU+메모리+네트워크 조합 이상 탐지 | 대기 |
| 2 | Prophet / statsforecast | 계절성 시계열 예측 고도화 | 대기 |
| 3 | Claude API / 로컬 LLM | 보고서 자연어 요약, 자연어 질의 | 대기 |

### 3순위 최우선 — 로그 수집 파이프라인 (Phase 0)

> **목적**: 자체 LLM 로그 분석 AI의 기반 데이터 파이프라인 구축
> GPU 불필요. 데이터가 쌓여야 미래에 파인튜닝 가능

#### 스코프 결정사항

| 항목 | 결정 |
|------|------|
| 로그 소스 | Linux: journald + `/var/log/*` 파일 tail · Windows: Event Log (Application/System) |
| 수집 주체 | 기존 OWLmon Agent 확장 (Vector/Fluent Bit 미사용, 단일 바이너리 유지) |
| 전송 프로토콜 | OTLP/HTTP Logs (gzip 배치, 기존 메트릭 전송 방식과 일관) |
| 저장소 | PostgreSQL + 일별 파티셔닝 (`pg_trgm` GIN 인덱스로 텍스트 검색) |
| 기본 보존 기간 | **90일** (ISMS-P 친화, config로 30/90/180/365 택일 가능) |
| 라벨링 데이터 | `log_annotations` 테이블 영구 보관 (미래 LLM 학습 데이터) |
| 민감정보 마스킹 | **기본 on** (이메일/주민번호/휴대폰/카드/IP 정규식) |
| 컨테이너 로그 | 제외 (journald가 stdout 컨테이너 로그 대부분 포착) |

#### 구현 범위 (6주 마일스톤)

- **Week 1**: DB 스키마 + 파티션 관리자 + `POST /v1/logs` 수신 + JWT
- **Week 2**: Agent journald + 파일 tail + WAL 버퍼 + OTLP 전송
- **Week 3**: 검색 API (`GET /api/logs`) + `pg_trgm` + 라벨링 API
- **Week 4**: React 로그 검색 UI + 호스트 상세 "로그" 탭 + AnnotateModal
- **Week 5**: Windows Event Log + Rate limit + 민감정보 마스킹
- **Week 6**: 부하 테스트 + 문서 + 납품 패키징

#### 완료 판정 (DoD)

- Agent 메모리 ≤ 15MB, CPU ≤ 1.5% (로그 수집 on 상태)
- 단일 서버 초당 5,000건 수신/저장 무손실
- 1천만 건 중 키워드 검색 P95 ≤ 1초
- Agent 재시작 시 WAL에서 미전송 로그 복구
- 이상 알림 1건 → "원인/조치" 라벨링 → `log_annotations` 저장 E2E

#### 스키마 요약

- `logs` — 파티션 테이블 (host_name, source, timestamp, severity, message, template_id, attributes JSONB)
- `log_annotations` — 영구 보관 (log_id, annotator, problem, solution, alert_id)

### 3순위 — 차별화 포인트

#### SSL 인증서 만료 알림
- **왜**: 학교 홈페이지 인증서 만료 = 민원. 담당자가 놓치는 경우 많음
- **내용**: 도메인 등록 → 만료 30일/7일 전 이메일 알림
- **구현**: crypto/tls로 인증서 조회, 에이전트 또는 서버에서 주기적 체크

---

### 4순위 — 완성도

#### 서버 추가 온보딩 UI
- **왜**: 현재 에이전트 설치가 수동. 납품 시 고객이 직접 못 함
- **내용**: 대시보드에서 에이전트 설치 스크립트 생성 / 안내

#### 자체 LLM 로그 분석 AI (장기 — Phase 1~5)
- **왜**: 로그 이상 → 원인/조치를 자동 해설. 공공기관 데이터 반출 불가 → 온프렘 파인튜닝 필수
- **전제**: Phase 0 로그 수집 파이프라인 완료, 운영자 라벨링 데이터 축적
- **단계**:

  | Phase | 내용 | GPU 필요 |
  |-------|------|---------|
  | 1 | Drain3 로그 템플릿화 + 템플릿 빈도 Z-score 이상탐지 | ❌ |
  | 2 | 운영자 피드백 UI (log_annotations 축적) | ❌ |
  | 3 | 규칙 기반 해설 라이브러리 (흔한 패턴 100개) | ❌ |
  | 4 | Claude API + RAG로 자연어 해설 (UX 검증용) | ❌ (API 과금만) |
  | 5 | Qwen 2.5 7B QLoRA 파인튜닝 → 증류 → 온프렘 소형 모델 배포 | ✅ |

- **체크포인트**: (이상→해결) 라벨 500건 이상 + RAG UX 검증 통과 시 Phase 5 착수
- **보너스**: 월간 보고서에도 같은 LLM으로 자연어 요약 생성 ("이번 달 서버 A는 매주 화요일 오전 CPU가 급등하는 패턴이 반복됐습니다…")
